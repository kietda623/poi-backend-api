    else Vượt quá giới hạn gian hàng
        API-->>Portal: 400 BadRequest {message:"Số gian hàng hiện tại vượt quá giới hạn gói"}
        Portal-->>Seller: ❌ Thông báo lỗi + gợi ý xóa bớt gian hàng hoặc chọn gói cao hơn
    end

    Note over Portal,API: Sau khi đăng ký thành công, mọi API call đều kiểm tra Subscription
    Seller->>Portal: Truy cập tính năng "Thống kê nâng cao"
    Portal->>API: GET /api/owner/shops/stats (Header: Authorization Bearer)
    API->>DB: SELECT s.Status FROM Subscriptions s WHERE s.UserId=? AND s.Status='Active' AND s.EndDate > NOW()
    DB-->>API: tier = "Premium"
    API->>API: CheckFeatureAccess("AdvancedStats", tier) → true
    API->>DB: SELECT usage stats...
    DB-->>API: StatsDto
    API-->>Portal: 200 OK + StatsDto

Phần 06

Mobile App — UsageHistory Sync v2

Cập nhật kiến trúc đồng bộ lịch sử trong .NET MAUI app. Thay vì gọi API ngay sau mỗi sự kiện nghe, Mobile lưu local và đồng bộ batch định kỳ, giúp tiết kiệm pin và băng thông.

🔄 Luồng Sync mới (Offline-first Batch)

Bước 1 — Ghi Local (ngay lập tức)

Sau khi kích hoạt thuyết minh, UsageHistoryService ghi bản ghi vào SQLite local với trường IsSynced = false. Không cần mạng, không block UI.

Bước 2 — Queue & Accumulate

Các bản ghi chưa đồng bộ tích lũy trong SQLite. Background Worker kiểm tra hàng đợi mỗi 5 phút hoặc khi kết nối mạng được phục hồi.

Bước 3 — Batch Upload

Gom tối đa 100 bản ghi gửi một request lên POST /api/usage-history/batch. Nếu thành công (HTTP 200), đánh dấu IsSynced = true và ghi SyncedAt.

Bước 4 — Retry Strategy

Nếu request thất bại: exponential backoff (5s → 30s → 2min). Sau 3 lần thất bại, bỏ qua batch hiện tại, thử lại ở chu kỳ tiếp theo. Bản ghi vẫn giữ nguyên trong SQLite.

Bước 5 — Cleanup

Bản ghi đã sync > 30 ngày tự động xóa khỏi SQLite để tiết kiệm bộ nhớ thiết bị.
